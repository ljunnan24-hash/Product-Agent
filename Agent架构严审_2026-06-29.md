# Agent 架构严审

日期：2026-06-29

## 结论

这次问题要按 P0 处理。当前项目不是没有 agent 架构，而是还没有把所有高上下文、高风险、长耗时任务强制纳入一个不可绕过的 agent runtime。

已有能力包括 `SubagentRunner`、`Subagent Registry`、`ContextManager`、`TaskGraph`、`GraphExecutor`、`Worker Queue`、`Durable Queue`、`Interrupt`、`Resume`、`Tool Security`、`Code Executor v1.6`、回测和盲测。但很多能力仍偏“账本和可见性”，还没有完全变成“执行控制平面”。

一句话：我们已经能看见 agent 做了什么，但还没有足够强地规定 agent 必须怎么做、怎么隔离、怎么恢复、怎么评估、怎么拒绝。

补充状态：API 调用、文件读写、本地 JSON 数据库、网页搜索/抓取、PDF/OCR/GitHub 导入已经是实际任务执行能力；代码执行已补 v1.6，能在受限 Python 或可选 Docker no-network 容器中汇总上传实验 CSV/JSON，生成 `summary_chart.svg`，并写入 artifact/handoff/trace。v1.6 已有静态危险模式拦截、路径约束、资源限制、输出审计、SVG 预览、MIME/SVG 安全审计、durable replay、运行中取消、cleanup audit 和 replay 后实验证据局部刷新；但默认部署仍可能降级到进程级受限执行，且目前还不是微虚拟机/远端隔离执行环境。独立 worker daemon 已有本地 drain、心跳/run log、状态 API、报告页面板、UI 启停、crash 自动拉起、lease 恢复展示，`code_execute`、`evidence_extract`、`judge` 和 `model_report` 已接入 durable replay/daemon drain 能力；已补 launchd plist 管理脚本、`worker:launchd`、API launchd 状态、health SLA 和 queue SLA/alerts 快照。但它还不是带生产级隔离和外部告警通道的 supervisor。

## 外部基线

### Claude Code / Claude Agent SDK

Claude Code 的 subagent 重点不是“多一个角色名”，而是每个 subagent 有独立 context window、自定义 system prompt、特定工具权限和独立权限边界。适合处理会淹没主上下文的搜索结果、日志和文件内容，只把摘要返回主会话。

对本项目的要求：

- 网页搜索、网页抓取、README/PDF 深读、证据抽取、反证、时效、竞品、Judge、Report 都必须走隔离 worker。
- 主 agent 只接收 handoff、evidence card、artifact ref，不接收长网页和原始搜索噪音。
- subagent 权限必须显式，不允许默认继承过宽工具权限。

参考：

- https://docs.anthropic.com/en/docs/claude-code/sub-agents
- https://code.claude.com/docs/en/agent-sdk/subagents
- https://code.claude.com/docs/en/tools-reference
- https://code.claude.com/docs/en/agent-sdk/permissions

### Anthropic Agent Patterns / Context Engineering

Anthropic 的有效 agent 模式包括 prompt chaining、routing、parallelization、orchestrator-workers、evaluator-optimizer。对调研型任务，orchestrator-workers 和 evaluator-optimizer 是关键：主控拆任务，worker 并行查证，独立 evaluator 压住结果质量。

Context engineering 的重点是 compaction、隔离和最小高信号上下文，而不是把更多原文塞进 prompt。

对本项目的要求：

- Product Agent 应该是 research agent，不是“上传材料后大 prompt 猜潜力”。
- 证据链路必须支持 fan-out / fan-in：支持、反证、时效、竞品并行，然后合并。
- Judge 和 AgentRunEval 要作为独立质量门，而不是报告后的装饰。

参考：

- https://www.anthropic.com/research/building-effective-agents
- https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

### OpenAI Agents SDK

OpenAI Agents SDK 把 agent 定义为带 instructions、tools、guardrails、handoffs、sessions 的运行单元；tracing 覆盖 LLM generation、tool call、handoff、guardrail 和 custom events；RunState 是 human-in-the-loop 的可序列化恢复边界；长任务可接入 Dapr、Temporal、Restate、DBOS 等 durable runtime。

对本项目的要求：

- trace 不只展示 UI，要成为调试、恢复、评估、回放的事实来源。
- 每个工具调用都要有 guardrail，而不能只靠 agent 输入/输出层 guardrail。
- long-running agent 需要 durable execution，不应依赖单个 HTTP 请求生命周期。

参考：

- https://developers.openai.com/api/docs/guides/agents
- https://openai.github.io/openai-agents-python/agents/
- https://openai.github.io/openai-agents-python/tracing/
- https://openai.github.io/openai-agents-python/guardrails/
- https://openai.github.io/openai-agents-python/ref/run_state/
- https://openai.github.io/openai-agents-python/running_agents/

### LangGraph

LangGraph 明确区分 short-term checkpoint、long-term store、interrupt。Checkpoint 用于恢复单次运行，store 用于长期记忆，interrupt 用于暂停图执行并等待外部输入。

对本项目的要求：

- Memory 不能替代 checkpoint。
- Interrupt 不是 error，而是正式执行状态。
- 运行中断后必须能从 node checkpoint 恢复，而不是整份分析重跑。

参考：

- https://docs.langchain.com/oss/python/langgraph/persistence
- https://docs.langchain.com/oss/python/langgraph/interrupts
- https://docs.langchain.com/oss/python/concepts/memory
- https://docs.langchain.com/oss/python/langchain/human-in-the-loop

### MCP / Tool Schema

MCP 的核心启发是外部工具和资源需要统一 schema、metadata 和边界。工具不是随便写一个函数让模型调，而是要有输入输出 schema、权限、风险、成本、超时、缓存、guardrail 和审计。

对本项目的要求：

- `web_search`、`web_fetch`、`file_read`、`pdf_extract`、`ocr`、`github_import`、`evidence_extract`、`judge`、`model_report`、`follow_up` 都必须进入统一 tool registry。
- 工具输出不能默认进入模型上下文，只能通过 context pack / artifact ref / handoff 传递。

参考：

- https://modelcontextprotocol.io/specification/2025-06-18
- https://modelcontextprotocol.io/specification/2025-06-18/server/tools

### Agent Security

Microsoft 2026-06-18 披露的 AutoJack 说明：一个恶意网页可以借助本地 browsing agent 跨过 localhost 信任边界，触达本地 MCP 控制面并造成 host-level RCE。OWASP 也把 Prompt Injection 列为 LLM 应用高风险，间接 prompt injection 可来自网页、文件、邮件等外部内容。

对本项目的要求：

- localhost 不是 agent 的安全边界。
- 网页、README、PDF、GitHub issue、截图 OCR 都是 untrusted evidence。
- webfetch 不能只做静态 URL 判定，还要考虑 redirect、DNS rebinding、内容注入、本地控制面和凭据泄漏。
- durable input payload、run log、analysis record 也要做 secret redaction。

参考：

- https://www.microsoft.com/en-us/security/blog/2026/06/18/autojack-single-page-rce-host-running-ai-agent/
- https://genai.owasp.org/llmrisk/llm01-prompt-injection/
- https://owasp.org/www-project-top-10-for-large-language-model-applications/

## 当前项目状态

### 已经做对的部分

1. 单入口对话和材料上传入口已经符合产品方向。
2. Web research 已有 query plan、搜索、抓正文、证据归一、反证、时效、竞品路径。
3. `SubagentRunner` 已让多类 worker 记录 worker run、context pack、transcript、budget、failure code。
4. `Subagent Registry` 已开始统一角色、工具 allowlist、模型/provider、memory scope、评估指标和安全注记。
5. `ContextManager` 已有 context boundary、payload preview、artifact ref 截断和 violation 记录。
6. `TaskGraph / GraphExecutor` 已有 node definition、依赖、priority、concurrency group、timeout、retry、interrupt、freshness。
7. `Durable Queue / Replay / Drain` 已能持久化并 replay `web_search` / `web_fetch`。
8. `Tool Security` 已集中处理 public URL、SSRF 初步阻断、prompt injection 信号、secret redaction。
9. `Hard Interrupt / Resume` 已有第一版，缺 key、Judge 阻断等可以进入 interrupt。
10. 回测、盲测、报告质检、证据绑定已经能帮助检查结果质量。

### 仍然遗漏或不够硬的部分

#### P0-1：Subagent 强制入口已完成第三段

现状：很多 worker 已接入 `SubagentRunner`。主分析 runtime 已前移到材料读取之前，`file_read`、`pdf_extract`、`github_import` 等材料工具现在真实运行在 `material-reader` runner 内。继续对话上传材料和实验原件回填也已迁入共享 supplemental material runner，统一通过 `material-reader` 运行 `file_read`、`pdf_extract`、`ocr`。follow-up Evidence Brief 重算已迁入 `evidence-extractor` worker。报告草案生成/应用已迁入 `report_composer` worker，记录 `model_report` tool call、artifact 和 handoff。AgentRunEval 已新增 `Runner Enforcement`，会检查高风险工具是否挂在 worker/task/boundary 下。

剩余风险：quality research 和部分派生产物刷新仍可能走过程式链路；`collectWebResearch` 对未预创建 runtime 的调用仍有兼容兜底，需要继续缩小绕过面。

下一段应补：

- quality research 继续迁入 registry + runner。
- 禁止业务函数直接把长材料、网页正文、搜索结果传给 Judge / Report。
- 每个 worker 必须产出 artifact + handoff，而不是大对象回传。

#### P0-2：GraphExecutor 还没有成为真正 durable graph runtime

现状：TaskGraph 和 worker queue 已同步很多状态，但图执行还不是完整的事件驱动 runtime。

风险：长任务仍可能被请求生命周期、开发服务器、局部 replay 粒度影响。

应补：

- task node input / output / error / lease / checkpoint 全部持久化。
- GraphExecutor 负责调度 ready nodes，而不是过程式函数推进。
- 每个 node 都支持 cancel、retry、requeue、timeout、resume。

#### P0-2b：Code Execution 还不是强沙箱

现状：`code_execute` 已作为高风险工具进入 `code_executor` subagent、tool policy、context pack、task node、artifact 和 Runner Enforcement。实验 CSV/TSV/JSON 上传会自动运行受限 Python 汇总，输出 `code_execution_result`、`summary.json`、`summary.md` 和 `summary_chart.svg`。当前已有代码长度、输入文件、`open()` 路径、输出文件数量/大小/总量、CPU、内存、文件大小和打开文件数限制，并已支持 durable queue/replay；replay 后如已有实验结果，会刷新 `experiment_code_execution` Evidence Card 和 Evidence Brief。v1.5 增加 `CODE_EXECUTOR_SANDBOX=auto|process|docker`，支持本地 Docker 镜像可用时用 `--network none` 容器执行；docker 严格模式不可用时阻断，auto 模式不可用时降级并记录 warning。v1.6 增加运行中取消和 cleanup audit：取消请求会杀 Python 进程或 Docker 容器，并把 durable record、runtime queue item 和 task node 写成 `cancelled`；超时也会杀进程并记录 cleanup guardrail。报告页已提供 worker queue 控制台，可对 queued/running 执行取消，对 failed/skipped 执行重排或重放。

风险：当前已有可选 Docker no-network 后端，但默认 auto 模式在 Docker daemon 或本地镜像不可用时会降级到进程级受限执行；这足以做小型数据计算验证，但还不是 Firecracker、gVisor 或远端隔离执行环境。

应补：

- 微虚拟机/远端隔离沙箱，或把 Docker 严格模式变成生产必选。
- CPU、内存、磁盘和文件数配额。
- 禁止网络的系统级策略在所有生产环境强制生效，而不只在 Docker 可用时生效。
- 更多图表类型和前端对比视图。
- code task node 的后台常驻 worker、重排策略和更细粒度派生产物刷新。

#### P0-3：AgentRunEval 已完成第一段

现状：已有 `agent-run-eval.ts`、`AgentRuntimeTrace.runEval` 和报告页/回测页展示。评估器不调用模型，确定性检查 task graph 覆盖、实际完成的支持/反证/时效/竞品路径、SubagentRunner 隔离、context boundary、tool guardrail/security、handoff 完整性、Judge/Report 边界、恢复能力、工具效率和 runtime versioning。旧分析和旧回测读取时会自动补算 eval。

剩余风险：当前还是 v1 轨迹规则，尚未把 source hash、fetch time、引用片段 checksum、证据过期、成本/latency 和 run manifest 纳入评分。

下一段应补：

- 把 Evidence Provenance / Freshness v2 纳入 eval。
- 把 cost ledger、latency、provider failure 和 cache 效率纳入 eval。
- 把 prompt/schema/tool policy/context policy/model/provider 版本写入 run manifest，并纳入 versioning check。

#### P0-4：Web evidence security 还不够

现状：已有 URL 安全和 prompt injection 初筛，但防线还偏薄。

风险：AutoJack 类问题说明浏览网页的 agent 可能成为外部网页和本地控制面的桥。

应补：

- fetch 前后都要按 redirect 后最终 URL 重新做 public URL 判定。
- 增加 DNS rebinding / IP resolve 后私网检测。
- durable input payload、worker daemon log、analysis record 全面 redaction。
- 给恶意 README / PDF / 网页做红队样本。
- 明确禁止 agent 从网页内容中学习任何操作指令。

#### P0-5：Memory Store 还只是概念

现状：registry 已有 memory scope，但没有正式 memory store。

风险：如果先把历史结论塞进上下文，会污染当前证据判断。

应补：

- 区分 run checkpoint、product memory、calibration memory、procedural memory。
- 每条 memory 必须有 source、createdAt、updatedAt、scope、confidence、TTL、conflict policy。
- memory 只能作为 hint，不能绕过当前证据标准。

#### P0-6：Evidence Provenance / Freshness 还不到可审计

现状：有 publishedAt、updatedAt、recencyBucket、Source Budget，但还缺严格核验字段。

风险：产品生命周期很敏感，过期证据会造成高置信错误判断。

应补：

- source hash、fetch time、final URL、redirect chain、content checksum。
- 引用片段 checksum、claim binding、date source。
- 按产品生命周期设置 evidence half-life 和 stale refresh。
- 冲突证据合并规则。

#### P1-1：Provider Router / Cost Ledger 不完整

现状：已有智谱、DeepSeek、搜索 provider 路由，但缺统一成本和质量账本。

应补：

- 每个 worker 记录模型、provider、tokens、搜索次数、抓取数、latency、失败率、cache hit。
- 根据任务选择模型：材料理解、搜索规划、证据抽取、Judge、Report 不应同一策略。
- 成本超预算时进入 interrupt 或降级路径。

#### P1-2：Versioning / Replay Ledger 不完整

现状：trace 有模型和工具局部记录，但无法完整回答“为什么这次回测变好/变坏”。

应补：

- 记录 prompt version、schema version、tool policy version、context policy version、judge rule version、model/provider version。
- 每次回测和分析保存 run manifest。
- 任何报告都能回放到当时的 harness 版本。

#### P1-3：Harness UI 还不是操作控制台

现状：页面能看到 task、worker、tool、boundary、interrupt，但还不够像 agent control plane。

应补：

- Supervisor -> task graph -> subagent -> tool -> artifact 的树。
- 实时 streaming event，而不是运行后静态列表。
- 每个节点可看输入摘要、输出 artifact、guardrail、失败原因、resume 按钮。
- 不展示隐藏思维链，只展示可审计状态、计划、证据、决策边界。

## 修正后的优先级

### 第一优先级：AgentRunEval v1 第一段已完成

原因：我们不能再靠“看起来过程很多”判断 agent 做得好不好。必须让每次运行自动回答：它有没有认真查证、有没有查反证、有没有考虑时效、有没有浪费工具、有没有在证据不足时停下来。

已交付：

- `AgentRunEval` 类型和确定性评分器。
- 接入 `AgentRuntimeTrace`。
- 报告页和回测页展示 score / blockers / warnings。

下一步：把 Subagent 强制入口、Evidence Provenance/Freshness 和 Tool Security v2 接进 eval，让评分不只看轨迹结构，也能看证据可核验性与时效。

### 第二优先级：Subagent 强制入口

原因：只要高上下文路径还能绕过 runner，context isolation 就不可靠。

已交付：

- 主分析材料读取真实运行在 `material-reader` worker 内。
- follow-up 上传材料和实验原件回填真实运行在 `material-reader` worker 内。
- follow-up Evidence Brief 重算真实运行在 `evidence-extractor` worker 内。
- 报告草案生成/应用真实运行在 `report_composer` worker 内。
- AgentRunEval 新增 Runner Enforcement runtime check。
- 报告页 eval 面板展示 enforced / unenforced。

下一段交付：

- quality research 迁入 registry + runner，并继续缩小 route 内过程式刷新范围。

### 第三优先级：Evidence Provenance / Freshness v2

原因：这个产品要减少主观判断，证据来源、抓取时间、发布日期和生命周期过期策略必须严肃。

交付：

- source hash、fetch time、final URL、date source、quote checksum。
- stale evidence evaluator。
- report citation verifier。

### 第四优先级：Tool Security v2

原因：AutoJack 说明 web-enabled agent 的安全边界比普通网页抓取更危险。

交付：

- redirect 后 URL 检测。
- DNS/IP 私网检测。
- durable payload / daemon log / analysis record redaction。
- 恶意网页/PDF/README 红队集。

### 第五优先级：Memory Store

原因：memory 很重要，但必须在 checkpoint / provenance / eval 之后做，否则旧结论会污染判断。

交付：

- Product / Calibration / Procedural memory schema。
- TTL、source、scope、conflict policy。
- 只作为 context hint，不作为证据。

## 对我前面遗漏的判断

这次我真正漏掉的不是“subagent 这个词”，而是没有一开始把 Product Agent 当成一个 research runtime 来设计。

普通 chat 产品可以先做对话体验；这个产品不行。它从第一天就涉及外部网页、用户上传材料、长期任务、证据时效、模型报告、可恢复执行和安全边界。因此它的地基应该是：

1. durable graph runtime
2. isolated subagents
3. strict context manager
4. tool policy and security
5. evidence provenance
6. interrupt / resume
7. trajectory eval
8. governed memory

后续不应该继续零散补 UI。AgentRunEval 第一段已经落地，下一步把 runner 做成不可绕过的硬入口，然后补证据来源和安全二层。这样项目才会从“能演示的 agent”往“能信任的 agent”走。
