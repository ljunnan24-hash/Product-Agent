# 大师如何判断一个产品

## 结论

高手判断产品，不是先问“好不好看”，也不是先问“功能多不多”。

他们通常按这个顺序判断：

1. 这个产品承诺了什么？
2. 它到底帮谁完成什么进步？
3. 这个问题够不够痛、够不够频繁、够不够值钱？
4. 用户会不会选择它，而不是继续用替代方案？
5. 用户能不能快速理解、相信并完成关键动作？
6. 产品是否能被商业、渠道、成本、品牌和组织现实支撑？
7. 是否有留存、真实使用、传播或付费信号？
8. 产品是否足够聚焦，把非必要内容都拿掉了？
9. README、官网或外部网页是否能证明用户真的需要它？
10. 下一步最该做什么，才能最快学到东西？

## 1. Marty Cagan：先看四大风险

参考：SVPG《The Four Big Risks》

核心判断：

- Value risk：用户是否真的想要，是否会选择使用或购买。
- Usability risk：用户是否能弄懂、能完成关键动作。
- Feasibility risk：团队是否能在现实约束下做出来。
- Business viability risk：商业、渠道、法律、成本、品牌、盈利是否成立。

对 Product Agent 的启发：

- 不应该只评价页面、文案或审美。
- 每次诊断都要至少覆盖 value / usability / viability。
- feasibility 在材料不足时标记为未知。

## 2. Clayton Christensen / JTBD：看用户想完成的进步

参考：Christensen Institute《Jobs to Be Done Theory》

核心判断：

- 用户不是购买功能，而是在特定情境下“雇佣”产品帮自己完成进步。
- 要判断功能、社会、情绪三层力量。
- 竞争对手不一定是同类产品，也可能是旧流程、Excel、Notion、人工服务、拖延或不做。

对 Product Agent 的启发：

- 读产品介绍 PDF 后，先抽取“用户任务”，不是先抽功能列表。
- 诊断时要问：用户现在用什么替代方案？为什么愿意换？

## 3. Nielsen Norman Group：看理解、控制、反馈和认知负担

参考：NN/g《10 Usability Heuristics》

核心判断：

- 用户需要知道系统状态。
- 产品语言要匹配用户真实世界，而不是内部术语。
- 用户需要控制感和可退出路径。
- 界面要减少记忆负担。
- 信息和视觉要聚焦在核心任务上。

对 Product Agent 的启发：

- 不是只看 UI 是否漂亮，而是看用户能不能理解下一步。
- 诊断报告必须指出“哪里让用户理解成本变高”。

## 4. Dieter Rams：看有用、可理解、诚实、克制

参考：Vitsoe《Ten principles for good design》

核心判断：

- 好设计让产品有用。
- 好设计让产品可理解。
- 好设计是诚实的，不夸大产品能力。
- 好设计尽可能少，但更好。

对 Product Agent 的启发：

- 产品页不能承诺超过产品真实能力。
- 视觉和文案应该放大核心用途，而不是制造装饰。

## 5. Brian Balfour：看产品市场匹配信号

参考：Brian Balfour《The Never Ending Road To Product Market Fit》

核心判断：

- PMF 不是一个瞬间，而是一系列检查点。
- 调研问卷只是 leading indicator。
- 更强信号来自真实 engagement、retention 和 meaningful usage。
- 没有留存，增长没有意义。

对 Product Agent 的启发：

- 如果材料只讲愿景，没有测试、使用、留存或付费信号，要指出这是关键缺口。
- 下一步行动要优先帮助用户获得更快的市场学习。

## 6. README 和网页证据

对早期软件产品，README 往往比精美页面更接近真实产品能力。

Product Agent 读取 README 时应该重点看：

- 项目一句话是否清楚。
- 安装和使用路径是否低摩擦。
- 目标用户和使用场景是否具体。
- 是否有 demo、文档、GitHub、官网、发布帖或用户反馈链接。
- README 是否解释了为什么不用现有替代方案。
- 是否有可信证据，例如截图、案例、stars、issues、用户 quote、waitlist、收入或活跃使用。

网页抓取和搜索的目的不是“显得 Agent 很忙”，而是补证据：

- 用户是否真的在搜这个问题。
- 竞品和替代方案是否存在。
- 社区是否有抱怨、讨论、教程或 hacky workflow。
- 产品是否有可触达的初始渠道。

## 7. Product Agent 的新判断链

Product Agent 的可见 workflow 应该是：

1. 接收材料
2. 读取 README / 产品介绍
3. 抓取 README 中的公开网页
4. 提取产品论点
5. 判断用户任务
6. 评估产品风险
7. 检查体验与信任
8. 判断市场与分发
9. 评估产品潜力
10. 匹配参考对象
11. 排序下一步
12. 生成诊断报告
13. 质量检查

## 8. 报告输出原则

报告不应该输出：

- “更高级一点”
- “更简洁一点”
- “视觉要更好”
- “建议多做用户调研”

报告应该输出：

- 哪个产品假设不清楚。
- 哪个用户任务没有被说透。
- 哪个风险最大。
- 哪个页面/材料部分影响理解或信任。
- 缺少什么证据。
- 外部网页证据支持什么、没有支持什么。
- 产品有没有潜力，以及成立条件是什么。
- 参考谁，具体学什么。
- 下一步应该先做哪 3 件事。
